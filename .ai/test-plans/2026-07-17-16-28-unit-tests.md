# Unit Test Plan: Environment Agent

## Overview

This plan identifies pure logic functions suitable for isolated unit testing in the DCM Environment Agent. Following strict BDD principles, we test **only** algorithmic behavior with no external dependencies — configuration parsing/validation, data structure operations (slot management, deny list), mathematical formulae (backoff), pattern matching (provider ID, topic name), state machine transitions (health), and structured data assembly (CloudEvents, RFC 7807 errors).

**Excluded from unit tests (covered by integration tests):**
- HTTP handler behavior (request/response lifecycle, middleware ordering)
- NATS/JetStream interaction (topic creation, consumer binding, ack/nack)
- REST client behavior (DCM registration calls, SP health poll calls)
- Persistence layer I/O (file read/write, corruption handling)
- Signal handling and graceful shutdown
- Logging output assertions (framework-mediated)

**Excluded by Caller-Trust Principle:**
- Validating that callers pass non-nil arguments where the type system prevents it
- Re-checking preconditions the router/framework already guarantees (e.g., method matching)

## 5. Retry Backoff Calculation

### UT-DCM-010: Exponential backoff formula without jitter

- **Validates AC:** AC-DCM-050
- **Component Under Test:** backoff calculator (deterministic component)
- **Given** `initialInterval=1s` and `maxInterval=5m`
- **When** calculating backoff for attempt 0
- **Then** the calculated interval (before jitter) MUST be `min(1s × 2^0, 5m) = 1s`
- **When** calculating for attempt 3
- **Then** the calculated interval MUST be `min(1s × 2^3, 5m) = 8s`

#### Boundary Cases
- UT-DCM-011: Attempt 0 → `1s` (no multiplication)
- UT-DCM-012: Attempt 1 → `2s`
- UT-DCM-013: Attempt 9 → `min(512s, 300s) = 300s` (capped at max)
- UT-DCM-014: Attempt 20 → still capped at `300s` (overflow-safe)

---

### UT-DCM-020: Full jitter produces value in valid range

- **Validates AC:** AC-DCM-050
- **Component Under Test:** backoff calculator (jitter application)
- **Given** `initialInterval=1s`, `maxInterval=5m`, and attempt=3 (calculated=8s)
- **When** jitter is applied (uniform random in [0, 8s])
- **Then** the result MUST be in range `[0, 8s]` inclusive

#### Boundary Cases
- UT-DCM-021: With deterministic random=0.0, result is 0
- UT-DCM-022: With deterministic random=1.0, result is the full calculated interval
- UT-DCM-023: With deterministic random=0.5 and calculated=8s, result is 4s

---

### UT-DCM-030: Retry-After header parsing (429 handling)

- **Validates AC:** AC-DCM-061
- **Component Under Test:** Retry-After parser
- **Given** a `Retry-After` header value of `"120"` (seconds)
- **When** the parser is invoked
- **Then** the result MUST be `120s`

#### Boundary Cases
- UT-DCM-031: HTTP-date format `"Thu, 01 Dec 2025 16:00:00 GMT"` — parsed correctly relative to current time
- UT-DCM-032: `"0"` — result is 0s (immediate retry)
- UT-DCM-033: Invalid value `"abc"` — falls back to standard backoff (returns sentinel/error)

---

### UT-RTE-050: Routing retry backoff uses same formula

- **Validates AC:** AC-RTE-055
- **Component Under Test:** backoff calculator (routing context)
- **Given** `retryBackoff=2s` and `retryMaxBackoff=30s`
- **When** calculating for attempt 4
- **Then** the calculated interval (before jitter) MUST be `min(2s × 2^4, 30s) = 30s` (capped)

#### Boundary Cases
- UT-RTE-051: Attempt 0 → `2s`
- UT-RTE-052: Attempt 3 → `min(16s, 30s) = 16s`
- UT-RTE-053: Attempt 10 → capped at `30s`
